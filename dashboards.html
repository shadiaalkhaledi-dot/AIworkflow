Practical tools

The Dashboards

Small, single-purpose web apps that solve a real, specific problem in my life. None of them are products — they're tools I built for an audience of one (or a small household, or one friend).

A quick note before the gallery: the BIM Coordination Studio below is a real tool running on a real, live project at work, so that one stays at the level of structure and workflow, not project specifics. Otherwise, treat each entry below less like "here's what Shadi built" and more like a small case study — what the problem was, what the tool actually does, and where the same pattern might be useful to you, even if you never touch a Google Sheet.

The pattern, in detail

Sheet → Script → Page → Link

Every dashboard below is a variation on the same four pieces. Here's what each one is actually doing.

PieceWhat it isWhat it's doing
Google SheetA normal spreadsheetHolds the actual data as rows — one row per listing, invoice, task, recipe, whatever the dashboard is about. Free, and Google already handles backups and access.
Apps ScriptGoogle's built-in scripting layer (JavaScript-based)Lives inside the sheet itself. Reads and writes rows, runs the logic ("flag anything over budget," "add this row only if it's not a duplicate"), and is the layer that talks to an AI when one is involved.
HTML pageA single web page (code, not a separate app)The actual thing you look at — cards, a kanban board, a map, charts. It calls the script for data and re-renders when something changes.
Web App deploymentOne menu action in the Apps Script editorTurns the page into a real URL you can open from a phone or bookmark, without paying for hosting or maintaining a server.
The one annoying gotcha

Editing the code doesn't automatically update the live link. You have to go back into the deployment settings and push a "new version" to the same deployment — otherwise you accidentally end up with a second, different URL. Small thing, but it trips people up the first time.

A real lesson learned

The moment a dashboard's link is set to "anyone with the link can open it," anyone can also try hitting the same endpoints the page itself uses to write data. The fix doesn't need to be fancy: a single shared password-like value gets checked on every write, and requests without it are rejected. It's not bank-grade security, but it stops a leaked link from turning into an open door — worth knowing even for tools nobody else will ever see.

What I've actually built

For each one: what the problem was, what the tool actually does, and a short note on where the same pattern could be useful to you — even somewhere completely different.

Apartment Search Tracker

Sheet + Apps Script + HTML

Built while apartment hunting in Calgary. Every listing — found manually or by Claude — lands in one place instead of scattered across browser tabs and texts to myself.

  • A kanban board (To Research → Contacted → Tour Scheduled → Applied → Offer) you drag cards across as things progress
  • A map view and a budget-aware color flag on every listing (under budget vs. over)
  • A scheduled task that runs every morning, searches multiple listing sites for new matches, skips anything already in the sheet, and adds verified new finds automatically
  • A per-listing activity log for call notes, tour impressions, and follow-ups, with overdue follow-ups flagged automatically

Useful for you if: you're ever comparing a lot of options against the same criteria — apartments, job postings, used cars, grad schools. Same pattern: one tracker, one set of filters, an automated scan so you're not re-checking the same five sites by hand every day.

BIM Coordination Studio

Apps Script + HTML — in daily use at work

The most involved thing on this page, and the one closest to my actual job. On a real building project, a dedicated clash-detection platform (Revizto, in my case) already finds every place two disciplines' models collide in 3D — a structural beam through a duct, a sprinkler line through a light fixture. What it doesn't manage is the human side: who owns each clash, what gets decided live versus over email, and whether anything's quietly gone stale. This dashboard is a managerial layer that sits on top of the clash engine — it never touches the 3D geometry, only the workflow around it.

  • A five-card live overview — Active, In Progress, Resolved, Accepted, Total — read straight off a pasted export from the clash engine
  • An issue register grouped by tag or priority, with a deep link back to each issue's real record
  • "Meeting Mode" — during the actual coordination meeting, walk through a group of related issues and bulk-decide them live on screen, instead of typing up minutes from memory afterward
  • Auto-generated pre-meeting briefs and post-meeting action items, both one-click printable PDFs sent to every discipline lead
  • Sprint-over-sprint trend tracking, with a backfill mode that can replay older exports to reconstruct history retroactively

Useful for you if: several people each own a slice of the same project and currently track open issues over email or marked-up PDFs. Swap "structural vs. mechanical" for whatever your own disciplines are — it's the same idea: one shared issue list instead of five separate inboxes, and a clean line between "the tool that finds problems" and "the system that makes sure they actually get resolved." The general pattern behind turning a written process like this into working software is walked through on the Pipelines page.

Meal Prep App

Apps Script + HTML + Gemini Live

Stores recipes in a structured format that separates "what I cook in bulk and freeze" from "what I do on the day I actually eat it" — including branching options (microwave vs. oven-finish) and side dishes timed against the main steps.

Useful for you if: you ever need to turn talking through something out loud into a clean written record, rather than typing it up later from memory — a site walk, a punch list, an inspection note, a verbal handoff to a contractor. The mechanic is identical to the recipe case; only the vocabulary changes.

Family Project Dashboard

Apps Script + HTML

A shared dashboard for tracking a family home project end to end — task checklist, budget categories, and timeline in one view, so everyone involved is looking at the same source of truth instead of separate group chats. (No real figures shown here — it's a household tool, not something built for outside eyes.)

Useful for you if: you're splitting a project with a few other people — a renovation, a trip, a group gift — and currently tracking it across three group chats and someone's notes app. One shared view kills most of the "wait, did we already pay for that?" messages.

Daily AI + AEC Brief

Scheduled task · Live artifact · Runs at 6:30 AM daily

This one's a different shape from the others — no Google Sheet, no web app URL. Instead: a scheduled task fires every morning at 6:30am, searches the web for the latest AI and AEC (Architecture, Engineering, Construction) news, and drops a structured brief into a live Cowork artifact before I've opened my laptop. The artifact keeps a 14-day rolling archive, and there's a "Run now" button for when something breaks mid-day and I don't want to wait until tomorrow.

  • Track 1 — AI & Tech: general AI developments that could affect how professionals work (new models, tools, policy, research)
  • Track 2 — AEC-specific: AI being used in architecture, engineering, construction, and BIM — firm pilots, software updates, papers, industry announcements
  • Each item: headline · short analysis · "why it matters for AEC" · source link
  • 14-day rolling archive, auto-pruned — no inbox, no newsletter, no manual curation

Useful for you if: you work in a field that's moving fast and want to stay current without checking five different sites every morning. Swap "AI and AEC" for whatever your own field is — the pattern is a scheduled search plus a standing artifact, not the specific topics.
