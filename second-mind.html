Knowledge system

The Second Mind

This is the one you specifically asked about. It's a personal knowledge base — built in an app called Obsidian — wired up to a grounded research tool and to Claude, so that things I learn don't just evaporate after I read them once.

The big idea

Two vaults: a workshop and a library

This is the vault I actually want to show you — the working system I run day to day with Claude, not the curated exam-prep one. It's split into two separate Obsidian vaults (think: two separate folders), and the split is deliberate.

Ops Vault

The workshop

Messy, on purpose. New information lands here first — half-finished research, AI-generated drafts, things copied from a PDF, ideas that might be wrong. Nothing in here is trusted yet, and that's fine, because nothing important is supposed to be retrieved from here long-term.

Main Vault

The library

Curated and trusted. A note only gets here after it's been checked against a real source. This is the vault I actually rely on day to day — if it's in here, I don't need to re-verify it again next time.

Why bother splitting them

Without the split, "things I'm pretty sure are true" and "things I've actually verified" end up mixed in the same pile, and over time you stop being able to tell them apart. Keeping the messy workshop physically separate from the trusted library means the trusted vault stays trustworthy — nothing gets in without going through the checkpoint first.

Useful for you, even without Obsidian

You don't need any of this specific tooling to borrow the idea. If you keep notes anywhere — Notion, Apple Notes, a stack of Google Docs — you probably already have one big pile where "I think this is true" and "I checked this" look identical. Just keeping a rough inbox separate from a small, deliberately curated "trusted" set of notes solves most of the actual problem. The plugins below just automate the moving part.

The 5 Promotion Rules

Before a draft note is allowed to move from the workshop to the trusted library, all five of these conditions must be met:

  • Evidence exists: The background context or data files actually exist.
  • Verified: The assertions are verified against primary references, not just guessed.
  • Reusable: The information is structured to be useful beyond the current conversation.
  • Correctly tagged: Saved in the right location with consistent YAML frontmatter.
  • Integrity: No information was distorted, lost, or hallucinated during the cleanup.
How to Split Into Two Vaults When Ready

You don't need two vaults on day one. But if you decide to fully separate your messy workshop from your trusted library, follow this sequence:

  1. Create the Second Vault

    Open Obsidian's vault switcher, choose "Create new vault," and point it to a completely empty folder. This becomes your trusted Main Vault.

  2. Enable Local REST API

    Go to Settings → Community Plugins → Browse, find "Local REST API with MCP", install, and enable it in the new vault, just like the first one.

  3. Set the Port (Avoid Collision)

    Open the plugin settings in the second vault. By default, it runs on port 27124. Change this to 27125 in the settings so the two vault connections don't collide.

  4. Register with Claude

    Copy the registration JSON block shown in the second vault's plugin settings page and ask Claude: "Register this second vault connection as 'trusted-vault' on port 27125." Claude will configure both connections so it can talk to both at the same time.

Intentional sessions

Pick a mode before you start

One of the most useful habits is starting every conversation with a single "mode" word. It acts like picking a mission: it immediately loads the relevant files, sets the target files for output, and switches Claude to the correct tone and rules.

Using a mode word saves token cost, prevents the AI from guessing, and helps keep your workspace cleanly segregated. Here are the modes I use:

Mode Word What it loads & focuses on Color Theme Mood & Tone
ARE Architecture Licensing Exam study tracker & resources Warm Sienna Rigorous, citation-conscious, study-oriented
BIM Learning model coordination & clash detection role resources Sage Teal Systematic, procedural, technical
Work Professional day-to-day coordination & project notes Slate Blue Efficient, highly structured, cautious
Finance Budget sheets & personal investment tracking (encrypted) Muted Gold Private, data-heavy, minimal chatting
Check-in A cross-cutting status sweep of all outstanding tasks Neutral High-level review, dashboard updates

Method 1: Simple Iconizing (No Code)

Install the Iconize community plugin in Obsidian. Right-click any mode folder in the sidebar, choose an icon, and pick its color. This is the fastest way to visually index your modes.

Method 2: Custom CSS Snippets

Create a CSS snippet in your vault's .obsidian/snippets/ folder. Set custom callout blocks (like >[!exam]) with specific color rules. This colors both the blocks inside your notes and the folder tabs in the sidebar.

The Signal of No Color

Leaving the general Check-in mode uncolored is a deliberate design choice. The absence of color serves as a visual signal that you are looking at a neutral, cross-cutting dashboard of your work rather than actively executing a specific project mission.

A third kind of dashboard

Dashboards that live entirely inside Obsidian — no spreadsheet at all

Everything on the Dashboards page is built on Google Sheets + Apps Script. This one is different: the home screen of this vault, a single note called Dashboard.md, is a fully live dashboard with no spreadsheet, no database, and no separate app behind it — just a Markdown note running real JavaScript every time it's opened, through a plugin called Dataview (specifically its "DataviewJS" mode).

Opening that one note, without me touching anything, shows:

  • A status table tracking two specific things side by side: my ARE prep (the architecture licensing exam) and a technical-skill transition at work, moving toward a BIM coordination role — model coordination, clash detection, that kind of thing. Each row shows sections or lessons completed, current streak, when I last touched it, and a specific next action ("second pass on practice scenarios," not just "study more")
  • A calendar heatmap (the same idea as a GitHub contribution graph) with both tracks overlaid in different colors on one calendar, so a day where I did both shows up as a visibly different third color — no need to open either tracker to see the pattern
  • Countdown timers to the next transition milestone, plus running streaks — current streak, best streak ever, this week's count
  • A full curriculum for the BIM side: roughly 113 lessons split across about a dozen domains (the relevant ISO standard, model coordination, data-delivery formats like COBie and IFC, the authoring software itself, team training, and a few others), each domain with its own built-vs-missing progress bar and a "study this next" card that always points at the next lesson that doesn't exist yet
  • Small inline forms — a dropdown, a text field, a button — that log a session by writing straight into that track's own note. No separate app opens; clicking the button edits the file directly and the dashboard re-renders on the next refresh

The same note also has a quick-capture panel built right into it — six small tabs (a coordination issue worth remembering, a research-tool query worth logging, a work day worth turning into portfolio evidence, a general learning note, a decision worth recording, an open question) each with two or three fields. Fill one out and hit submit, and it writes a new, properly-formatted note straight into the inbox — the same Evidence Capture pipeline described on the Pipelines page, just with the capture step happening right inside the dashboard instead of a separate app.

In plain terms

It's the spreadsheet pattern again, minus the spreadsheet — the note is the input form, the chart, and the to-do list, all at once. The "formulas" are real JavaScript that can read and write any file in the vault, which is something a spreadsheet formula can't do.

Why track two unrelated things in one place

They're less unrelated than they look. Hands-on BIM coordination work at the office doubles as documented practical experience toward the hours required for architecture licensure — so a note captured for one purpose can feed two different tracking systems at once. Seeing both side by side on one screen is what makes that overlap visible instead of theoretical.

Useful for you

You don't need my exact setup to borrow the idea. Any goal you're tracking daily or weekly — a language, a job search, the gym — can get the same treatment with just Dataview and a plain note: a streak counter, a "what's next" pointer, and a one-line log form, instead of yet another habit app you have to remember to open.

If you want to try this yourself

Start with just Dataview and Templater — that's the minimum for any of this to function. Add QuickAdd and Tasks once you're capturing things regularly. Everything else is polish, and you can add it whenever something feels missing rather than installing all fourteen on day one.

What I deliberately skipped

No Obsidian Git plugin — sync is handled more simply (file history / manual backup), and adding git just for a personal vault is more complexity than it's worth. No Calendar/Periodic Notes plugin — the daily-log folder structure already covers that. No Excalidraw — not needed for this workflow. And no AI plugins running inside Obsidian itself — Claude already does that job from outside the vault, so stacking another AI layer on top would just be redundant.

Connecting it to Claude

How to give Claude access to your vault

The plugins table above lists everything worth installing — but once Local REST API is in, the question is how to actually wire it up to Claude. There are two ways, and the simpler one is what I actually use day to day.

The key thing to know first

Obsidian stores everything as plain .md files in a regular folder on your computer. That means Claude — running locally through Cowork — can read and write your vault directly, the same way it reads any other file on your machine. You don't need a plugin or an API connection to get started. You just need to point Cowork at the right folder.

What I actually use

Direct folder access

Open Cowork, start a new project, and select your vault folder when prompted. Claude now has direct read/write access to every note. No plugin required. Works immediately.

First session: tell Claude your folder structure, which folders are drafts vs. trusted, anything else it needs to know. Ask it to save that as CLAUDE.md at the root — it reads that file automatically every future session.

Optional upgrade

Local REST API connection

Once Local REST API is installed and running, copy the MCP config block from its settings page (it's pre-filled with your port and token) and paste it into a Cowork session: "Register this MCP connection for me."

Adds structured search across tags and properties. Worth doing once you're comfortable with basics — not required on day one.

First prompt — let Claude interview you

Instead of writing your own CLAUDE.md from scratch, paste this in your first connected session:

Interview me — ask what you need to know about who I am, what I'm working on, and how I like to work. Once done, save the result as CLAUDE.md at the root of my vault.

Claude will ask you a series of questions, then write the profile. Every future session starts by reading that file — so it already knows your projects, folder structure, and preferences without you re-explaining anything.
