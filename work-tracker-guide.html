0 Download the starter kit and set up your workspace Do this first

One ZIP, five folders and a VERSION stamp. Each folder goes to a different place — the steps below explain exactly where. Nothing gets installed in the traditional sense; things mostly get copied and pointed at.

⬇ Download Starter Kit (.zip)

Inside: vault-setup/ · github-pages/ · apps-script/ · skills/ · tools/ · VERSION (currently 2026-07-01.2)

One prerequisite: the Claude desktop app (Cowork mode). That's what lets Claude actually touch files on your computer instead of just talking about them wistfully. Grab it at claude.ai/download if you haven't — everything below assumes it's installed.

📁 vault-setup/ — your working folder

This is the folder Claude reads and writes every session. The recommended setup is an Obsidian vault — Obsidian is a free note-taking app (obsidian.md) that stores everything as plain text files in a folder you own. When you create a new vault, Obsidian creates the folder for you. You then copy the contents of vault-setup/ into it.

Install Obsidian and create a new vault

Download Obsidian from obsidian.md. Open it and create a new vault — give it a name like "Work" or your name. Obsidian creates a folder on your computer automatically.

Copy the contents of vault-setup/ into your vault

From the ZIP, open vault-setup/. Copy everything inside — TASKS.md, index.html, and data.json — directly into your vault folder. These are the files Claude will read and update each session.

Connect the vault to Cowork

In the Claude desktop app, open Cowork and create a new project. When it asks for a folder, point it at your Obsidian vault. Cowork will now be able to read and write files there directly — no copy-pasting needed.

Don't want Obsidian? Then don't have it — I'm a narrator, not a salesman. Create a new Cowork project directly and Cowork makes the folder for you; copy the vault-setup/ contents in there instead. You lose the pretty note-taking layer, the tracker doesn't notice or care. It works exactly the same.

📁 github-pages/ — your live tracker website

index.html and data.json — the same two files that are also in your vault. Upload these to a free GitHub Pages repo so you can open your tracker from any browser. Step 2 walks you through this.

📁 apps-script/ — Quick Capture backend

One file: code.gs. There is no HTML to deploy here — this is a pure data API. You paste the code into Google Sheets Apps Script, deploy it as a web app, and Google gives you a URL. That URL goes into the "apiUrl" field in your data.json so the tracker knows where to send captures. Step 6 walks through this step by step.

📁 skills/ — the pipeline reference

One file: work-tracker-pipeline-friend.skill — the full architecture reference for this system: the three layers (vault → Apps Script → GitHub Pages), the exact tagging conventions, and the logic that turns your task list into data.json. Drag it into your first Claude chat and say "install this skill" (see Skills for the full install steps) so Claude has this as a standing reference in every future session — not just this one.

1 What's in the tracker — the two files explained

You already have both from Step 0 — but you should know what you're holding before you deploy it. Meet the skeleton and the heartbeat.

📄 index.html — the permanent shell

The skeleton. Filters, program view, stat tiles, quick capture, the password gate — all of it, and none of your data. Upload it to GitHub Pages once and never touch it again. It's the part of the tracker that doesn't know anything about you; it just knows how to display whatever the data file hands it.

🗂 data.json — your session data

The heartbeat. Claude rewrites this file every session with your current work — programs, open items, priorities, what got finished. Push a new version to GitHub and your live tracker updates in about thirty seconds. And once you've set a password, this file travels in disguise: what actually gets uploaded is data.enc.json, its encrypted twin, while the readable original never leaves your computer.

2 Put them on a free website (GitHub Pages)

GitHub Pages: a free, permanent URL for your tracker, from the same place that hosts half the internet's code. No coding involved — you'll mostly be clicking buttons with reassuring names.

Create a free GitHub account

Go to github.com and sign up. Use your personal email — your account stays with you even if you change jobs.

Create a new repository

Click the + in the top right → New repository. Name it something like work-tracker. Set it to Public. Click Create repository.

Upload your two files

On the repository page, click Add file → Upload files. Drag in both index.html and data.json. Scroll down and click Commit changes.

Enable GitHub Pages

Go to Settings → Pages (left sidebar). Under "Source", select Deploy from a branch, choose main, folder / (root). Click Save.

Get your tracker URL

After ~30 seconds, refresh the Pages settings. Your URL will appear — something like yourusername.github.io/work-tracker. Open it. Your tracker is live.

Updating later: Every session you replace one file and one file only. Repository → click the data file → pencil icon (Edit) → paste the new content → Commit. Thirty seconds later the live site matches reality. That's the entire maintenance burden of this system. I've seen houseplants that need more.

🔒 Protect your tracker with a password Recommended

GitHub Pages sites are public — anyone with the URL can read them, and making the repo private just unpublishes the site on a free plan. The kit solves this with real encryption instead.

Pick a password — yours, not anyone else's

Anything you'll remember. It never leaves your machine except as the key that unlocks the page in your own browser. If you ever forget it, nothing is lost — the plain data.json still lives in your folder; you just re-encrypt with a new password.

Ask Claude to encrypt your data file

The kit's tools/encrypt_data.py turns data.json into data.enc.json (AES-256 encryption — the real thing, not a hidden-field trick). Say: "Encrypt my data.json into data.enc.json using encrypt_data.py with the password ______".

Upload data.enc.json — and never the plain data.json

From now on the repo holds index.html + data.enc.json only. If a plain data.json is ever in the repo, delete it — it would defeat the whole gate. The plain file stays on your computer as the working copy.

Open your site

You'll see a password box and nothing else. Enter your password once — your browser remembers it after that. Visitors without the password see an empty page. Bonus: your Apps Script URL is inside the encrypted file, so even your capture endpoint stays hidden.

Before a password is set, the shell quietly falls back to reading the plain data.json — so everything in the steps below works out of the box, and the gate simply switches on the first time a data.enc.json appears in your repo.

3 First session — let Claude interview you Do this first

Do not fill in data.json by hand — that's a job for the AI, and frankly it enjoys it. Paste the prompt below and let Claude interview you; you talk, it builds.

Important — don't attach the file to the chat. Give Claude the folder path instead. It can read files right where they live, which means it can also update them right where they live — no downloading, re-uploading, or "wait, which version is this" archaeology later.

What happens next: Claude asks questions one at a time; you answer like a human — no formatting, no bullet points, no performance. Ten to fifteen minutes of normal conversation later, out comes a complete data.json. Save it, upload it, and your tracker is born already knowing your whole workload. Most trackers start empty and die empty. Not this one.

4 Every session after that — the check-in prompt

The daily ritual — and the whole reason this system stays alive while app-based trackers gather guilt. Paste this at the start of each session; Claude reads the current state, asks what changed, hands back the update.

📋 Copy this prompt — use it at the start of every work session

Work session check-in. My tracker folder is at:
[YOUR FOLDER PATH — e.g. C:\Users\yourname\Documents\Work Tracker\]

My live tracker site is at: [YOUR GITHUB PAGES URL]

If you have work-tracker-pipeline-friend.skill installed or available in this folder, use it as your reference for the tagging conventions (status/court/horizon/action/cat/disc) so things stay consistent as this grows.

Please read the current data.json and then ask me:
- What happened since last time?
- Anything new that's blocking me?
- Any items I can mark done?
- Any new items to add?
- Any meetings or decisions to capture?

After I fill you in, update data.json to reflect today's state — new items, completed items, updated notes, and a "what we sorted out today" summary. Set meta.generated to today's date.

Then, if I've set a site password: re-encrypt with tools/encrypt_data.py (data.json → data.enc.json) and give me data.enc.json to upload to GitHub — never the plain data.json. If I haven't set a password yet, give me data.json as before.
The rhythm: One check-in at the start of each day, or whenever you sit down to actually work. Five minutes, conversational, done. The site updates the moment you push the new file. Miss a day? Nothing breaks, nothing nags — the tracker just waits. It's the only productivity tool you'll ever own that doesn't guilt-trip.

5 Bringing in Copilot and email notes

Already drowning in Teams meetings and Copilot transcripts? Good news — that's not a problem, that's an input. Those notes feed straight into the tracker.

After a Copilot meeting summary

Copy the Copilot summary from Teams. Paste it into your check-in conversation with Claude and say: "Here's the Copilot summary from today's meeting — pull any action items or decisions into my tracker."

From email threads

Copy the relevant email thread or paste key lines. Say: "Add this to the relevant item in my tracker as a note." Claude will file it under the right item and program.

From a day of scattered notes

If you jot notes in Teams chat or a notepad during the day, paste them all at once during your check-in. Say: "Here's everything that happened today — sort it into my tracker." Claude knows where things belong.

Tip: The tracker page itself has a Quick Capture box (the gold one at the top) — thumb something in from your phone mid-meeting and it queues up for your next check-in. Five capture types: New item, Note, Blocker, Watch, and Subtask. Because the best time to record a thought is the four seconds before the next one deletes it.

6 Quick Capture — log things in real time Optional

The tracker's ears. A one-tap capture panel on your live page that files everything into a Google Sheet until your next check-in — for the thoughts that arrive mid-meeting, when opening a chat window would be rude and a sticky note would be a lie.

Full honesty: this is the most technical step on the page. It involves Google Apps Script, which sounds scarier than it is — but if it's not your day for it, skip it guilt-free. The check-in workflow already gives you a complete system. This is the optional turbo; come back when the basics are humming.
Create a Google Sheet

Go to sheets.google.com → New blank sheet. Name it Work Tracker Captures. You don't need to add headers by hand — the Apps Script below creates a "Log" tab with the right columns automatically the first time anything is captured.

Open Apps Script

In the sheet, go to Extensions → Apps Script. A code editor will open. Delete everything in the editor and paste the script below. Click the save icon (or Ctrl+S).

Deploy as a web app

Click Deploy → New deployment. Select type Web app. Set "Execute as" to Me and "Who has access" to Anyone. Click Deploy. Copy the long URL it gives you — this is your Apps Script URL. Keep it handy for the next step.

Paste the URL into data.json — this is how the tracker finds your sheet

Open data.json in your Cowork folder. Find the first line: "apiUrl": "PASTE_YOUR_APPS_SCRIPT_URL_HERE". Replace the placeholder with your real URL. Save, then push the updated data.json to GitHub. The tracker now knows where to send captures — no other file needs to change.

Tell Claude to pull captures at check-in

At your next check-in, say: "Also check for any pending Quick Captures from my sheet." Claude reads the unsynced rows (via the apiUrl in data.json), folds them into your task list, then marks them synced so they don't get pulled twice.

📋 Apps Script code — paste this into Extensions → Apps Script

var LOG_HEADER = ['Timestamp', 'Type', 'Program', 'Package', 'ItemRef', 'Text', 'Synced'];

function getSheet() {
  var ss = SpreadsheetApp.getActiveSpreadsheet();
  var sheet = ss.getSheetByName('Log');
  if (!sheet) {
    sheet = ss.insertSheet('Log');
    sheet.appendRow(LOG_HEADER);
    sheet.setFrozenRows(1);
  }
  return sheet;
}

function doPost(e) {
  var sheet = getSheet();
  var data = JSON.parse(e.postData.contents);

  if (data.action === 'edit') {
    return handleEdit(sheet, data);
  }

  sheet.appendRow([
    new Date(), data.type || '', data.program || '', data.pkg || '',
    data.itemRef || '', data.text || '', 'N'
  ]);
  return ContentService.createTextOutput(JSON.stringify({ok: true}))
    .setMimeType(ContentService.MimeType.JSON);
}

function handleEdit(sheet, data) {
  var row = Number(data.row);
  if (!row || row < 2 || row > sheet.getLastRow()) {
    return ContentService.createTextOutput(JSON.stringify({ok: false, error: 'Invalid row'}))
      .setMimeType(ContentService.MimeType.JSON);
  }
  sheet.getRange(row, 2, 1, 5).setValues([[
    data.type || '', data.program || '', data.pkg || '', data.itemRef || '', data.text || ''
  ]]);
  return ContentService.createTextOutput(JSON.stringify({ok: true}))
    .setMimeType(ContentService.MimeType.JSON);
}

function doGet(e) {
  var sheet = getSheet();
  var action = e.parameter.action || 'pending';

  if (action === 'markSynced') {
    var rows = JSON.parse(e.parameter.rows || '[]');
    rows.forEach(function(r){ sheet.getRange(r, 7).setValue('Y'); });
    return ContentService.createTextOutput(JSON.stringify({ok: true, marked: rows.length}))
      .setMimeType(ContentService.MimeType.JSON);
  }

  var values = sheet.getDataRange().getValues();
  var out = [];
  for (var i = 1; i < values.length; i++) {
    if (String(values[i][6]) === 'N') {
      out.push({row: i+1, ts: values[i][0], type: values[i][1],
        program: values[i][2], pkg: values[i][3], itemRef: values[i][4], text: values[i][5]});
    }
  }
  return ContentService.createTextOutput(JSON.stringify(out))
    .setMimeType(ContentService.MimeType.JSON);
}

function out(data) {
  return ContentService.createTextOutput(JSON.stringify(data))
    .setMimeType(ContentService.MimeType.JSON);
}
After pasting, authorize it: Google will ask you to approve the script's access to your own spreadsheet — click "Review permissions" → choose your account → "Allow". It looks alarming; it's Google asking whether your code may talk to your sheet. It may.

Field reference

FieldWhat it means
status"open", "blocking", "unverified", or "watch"
courtWho it's waiting on — "You", or a person's name
delegateOptional — if you've handed this off to someone, their name. Shows as "Delegated to: [name]" next to court on the card. Leave the field out entirely if nobody's been delegated to.
horizonUrgency — "today", "sprint" (before sprint end date), "week", or leave blank
actionWhat kind of task — "model", "review", "coordinate", "chase", "admin"
catCategory — "decisions", "meetings", "coordination", "documentation", "drawings", "rfi"
discDiscipline tags — "AR", "ST", "ME", "EL", "signage", "safety", "security", "owner"
isNewtrue = shows an N badge (new since last check-in). Set to false after first upload.
dueDue date in YYYY-MM-DD format — drives the stat tiles and urgency sort
Advanced — tag these fields directly in your task list: instead of letting Claude guess status/category/discipline/etc. each check-in, you can write them right into a task line as bracketed tags, e.g. [cat:: coordination] [disc:: AR] [action:: chase] [court:: You] [horizon:: today] [delegate:: Jordan]. Claude reads these mechanically instead of re-inferring them, which keeps your tracker consistent as it grows. See the example rows already in your TASKS.md. Not required — Claude can still triage without them — but the more you tag, the less guesswork there is.
